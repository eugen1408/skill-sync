## Обзор

Сейчас при старте (Часть 4, `SkillRegistry.recomputeOrphans`) любой установленный skill, для которого нет подключённого источника, попадает в «осиротевший» бакет `sourceId='installed'`, `sourceType='local'`. На чистой машине это означает, что все установленные skills пользователя (в примере — ~30 записей из `~/.agents/.skill-lock.json`) показываются как «локальные», хотя lock-файл знает их реальное происхождение (GitHub / произвольный Git / skills.sh).

Часть 8 использует `.skill-lock.json` как источник истины о происхождении установленных skills: при инициализации восстанавливает соответствующие источники (добавляет их сразу) и атрибутирует каждый установленный skill к правильному источнику. Признак «local» остаётся только за skills, которых **нет** в lock-файле.

## Пользовательская история

Как пользователь, впервые открывший приложение с уже установленными через CLI skills, я хочу видеть их сгруппированными по реальным источникам (skills.sh, GitHub-репозитории, произвольные Git-репозитории), а не свалку «локальных», чтобы понимать откуда что установлено и корректно проверять обновления.

## Критерии приемки

- [x] КП1: При инициализации приложение читает глобальный `.skill-lock.json` (`globalLockPath`, Часть 3) и использует его записи для атрибуции установленных skills.
- [x] КП2: Для каждой уникальной записи-источника из lock создаётся/переиспользуется подключённый источник (без дублей); источники добавляются сразу при инициализации.
- [x] КП3: Классификация источника lock-записи:
  - SSH-URL (`git@…` / `ssh://…`), не-github хост, либо `sourceType = git` → **custom git** источник (`type: 'git'`).
  - `sourceType = github` (https, github.com) → проверка принадлежности репозитория каталогу **skills.sh**: найдено → атрибуция к официальному источнику (`OFFICIAL_SOURCE_ID`); не найдено → **custom git** источник.
- [x] КП4: Установленный skill, присутствующий в lock, отображается записью каталога с `sourceId`/`sourceType`/`sourceRef` соответствующего источника (а не как `installed`/`local`), в т.ч. до индексации этого источника.
- [x] КП5: Только skill, **отсутствующий** в lock (нет записи по его нормализованному имени), считается локальным осиротевшим (`sourceId='installed'`, `sourceType='local'`).
- [x] КП6: Дедупликация источников: несколько skills из одного репозитория (напр. `anthropics/skills` → `skill-creator` + `pptx`) дают один Git-источник; skills.sh — единственный официальный источник.
- [x] КП7: Классификация/добавление источников не ломает быстрый старт: атрибуция не требует клонирования/индексации git-источников (индексация — по требованию / на следующем refresh).
- [x] КП8: Обработка недоступности skills.sh при старте — предсказуемое поведение (см. Q8-02), без падения инициализации и без потери установленных записей.

## Поведение

### Затрагиваемые компоненты / модули

- **`src/main/registry/lockAttribution.ts`** (новый) — читает глобальный lock, группирует записи по источнику (`sourceUrl`), классифицирует каждый уникальный источник (custom git vs skills.sh) и формирует:
  1. перечень источников, которые нужно гарантировать (git-источники: `url`, `ref`, `authMode`, `name`);
  2. карту атрибуции `нормализованное имя skill → { sourceKind, sourceUrl | 'official', sourceRef }`.
- **`src/main/sources/officialCatalog.ts`** (или соседний модуль-lookup) — метод проверки принадлежности репозитория каталогу skills.sh по `owner/repo` (+ имя/slug). Переиспользует существующий доступ к `skills.sh` (текущий `/api/search`; см. Q8-03).
- **`src/main/sources/SourceManager.ts`** — используется как есть (`add`, `list`, `ensureDefaultOfficial`, `assertNotDuplicate` уже дедуплицируют git по `url`+`ref` и запрещают дубль official). Возможен вспомогательный метод «добавить, если отсутствует» (idempotent add) поверх существующего `add`.
- **`src/main/registry/SkillRegistry.ts`** — на этапе построения индекса шаг `recomputeOrphans` заменяется/расширяется до `attributeInstalled`: установленный skill без покрытия индексированным источником атрибутируется по карте (Часть 8) к git/official-источнику; при отсутствии lock-записи — остаётся локальным осиротевшим (текущее поведение).
- **`src/main/index.ts`** — при старте, после `ensureDefaultOfficial()` и до/вокруг `skillRegistry.init()`, выполняется сидинг источников из lock (добавление недостающих) и передача карты атрибуции в `SkillRegistry`.

### Логика классификации (per-source, не per-skill)

Записи lock группируются по `sourceUrl` — классифицируется каждый уникальный репозиторий один раз (минимизирует сетевые вызовы):

1. `sourceType = git` **или** URL начинается с `git@`/`ssh://` **или** хост не `github.com` → **custom git**.
2. Иначе (`sourceType = github`, https, github.com) → запрос к skills.sh: репозиторий (`owner/repo` из lock-поля `source`) присутствует в каталоге?
   - Да → **official** (атрибуция к `OFFICIAL_SOURCE_ID`).
   - Нет → **custom git**.
3. Инвариант для этапа построения индекса: отсутствие git-источника для github-lock-записи ⟹ запись классифицирована как official (см. Q8-02 про поведение оффлайн).

### Логика атрибуции (в `SkillRegistry`)

Для установленного skill (по нормализованному имени), не покрытого записью индексированного источника:

- Есть lock-запись?
  - Найден подключённый git-источник с совпадающим `sourceUrl` → атрибуция к нему (`type: 'git'`).
  - Иначе, если lock-запись github/official-типа → атрибуция к официальному источнику (`OFFICIAL_SOURCE_ID`, `type: 'official'`, `sourceRef = "owner/repo@slug"`).
  - Иначе (`sourceType = local` в lock и т.п.) → локальный осиротевший.
- Нет lock-записи → локальный осиротевший (`sourceId='installed'`, `sourceType='local'`).

Запись каталога, созданная атрибуцией: `installed=true`, `installations[]` из `installedScanner`, `updateStatus='unknown'` до первой проверки версий, `sourceRef` — для git из lock (`source`/`skillPath`), для official — `owner/repo@slug`.

### Навигация

- Диплинки — не требуются.

## Требования к данным и API

### Читаемые поля `.skill-lock.json` (запись `LockEntry`, Часть 3)

```json
{
  "source": "string (owner/repo для github ИЛИ полный git-URL)",
  "sourceType": "string (github | git | local)",
  "sourceUrl": "string (URL для клонирования)",
  "skillPath": "string (путь до SKILL.md внутри репозитория)",
  "ref": "string | null"
}
```

### Результат классификатора (`lockAttribution`)

```json
{
  "sourcesToEnsure": [
    {
      "type": "string (git)",
      "url": "string",
      "ref": "string | null",
      "authMode": "string (https | ssh)",
      "name": "string (basename репозитория)"
    }
  ],
  "attribution": {
    "<normalizedSkillName>": {
      "sourceKind": "string (git | official)",
      "sourceUrl": "string | null (для git; null для official)",
      "sourceRef": "string (owner/repo@slug для official; идентификатор внутри репо для git)"
    }
  }
}
```

### Проверка принадлежности skills.sh

- Вход: `owner/repo` (+ имя/slug skill). Выход: `boolean` (репозиторий обслуживается каталогом skills.sh).
- Контракт реализации: через существующий доступ к skills.sh; сопоставление `source` (owner/repo) результата с lock-полем `source` (регистронезависимо). Точность зависит от доступного API (см. Q8-03).

## Пограничные случаи и обработка ошибок

1. **skills.sh недоступен при старте** — классификация github-записей невозможна; поведение по Q8-02, без падения инициализации.
2. **Источник уже подключён** (пользователь добавил его руками ранее) — не дублировать (`assertNotDuplicate` по `url`+`ref`); атрибутировать к существующему.
3. **Несколько skills из одного репозитория** — один git-источник, несколько записей каталога (КП6).
4. **Приватный репозиторий** — источник добавляется по метаданным lock; клонирование/индексация опирается на системный git при последующем refresh (не при старте).
5. **Skill есть в lock, но не найден на диске** (`installedScanner` не видит) — записи не создаётся (атрибутируются только реально установленные).
6. **`sourceType = local` в lock** — трактуется как локальный (не создаёт git/official-источник); см. Q8-01.
7. **Битая/неполная lock-запись** (нет `sourceUrl`) — запись пропускается для атрибуции, skill трактуется как локальный осиротевший; ошибка логируется, старт не прерывается.
8. **Расхождение имени skill и slug каталога** — нормализация ключа (`normalizeSkillKey`, Часть 4) при сопоставлении установленного с lock-записью.

## Открытые вопросы

## Q8-01: Обработка `sourceType = local` / отсутствующего `sourceUrl` в lock
**Приоритет:** Среднее
**Вопрос:** Как трактовать lock-записи с `sourceType = local` или без `sourceUrl`?
**Варианты:** 1. Трактовать как локальный осиротевший (не создавать источник). 2. Создавать local-источник по пути установки.
**Допущение:** Вариант 1 — локальный осиротевший (текущее поведение для «local»).
**Статус:** Отвечен
**Ответ:** Вариант 1 — локальный осиротевший.

## Q8-02: Поведение при недоступности skills.sh во время инициализации
**Приоритет:** Высокое
**Вопрос:** Если проверку принадлежности skills.sh для github-записи выполнить не удалось (сеть/лимит), к чему атрибутировать?
**Варианты:**
1. К официальному источнику (skills.sh) как наиболее вероятному для github; git-источник не создавать; переклассификация — future.
2. К custom git (создать git-источник из `sourceUrl`) — строго по правилу «не подтверждено найдено → custom git».
3. Отложить атрибуцию (оставить локальным осиротевшим) до следующего онлайн-старта.
**Допущение:** Вариант 1 — при недоступности skills.sh github-записи считаются official; git-источники не плодятся. Переклассификация при последующем онлайн-старте — future.
**Статус:** Отвечен
**Ответ:** Вариант 1 — считаем official. Но если по такой записи впоследствии возникает ошибка (напр. проверка версии/резолв не находит skill в skills.sh), запись переклассифицируется («свапается») в локальную осиротевшую (`sourceId='installed'`, `sourceType='local'`).

## Q8-03: API skills.sh для точной проверки принадлежности
**Приоритет:** Среднее
**Вопрос:** Достаточно ли `/api/search` (нечёткий поиск по строке) для надёжной проверки «репозиторий `owner/repo` есть в каталоге», или нужен точечный endpoint (по slug/owner/repo)?
**Допущение:** На первой версии — через `/api/search` по имени skill с последующим сопоставлением `source`==`owner/repo`; при недостаточной точности — уточнить endpoint позже.
**Статус:** Отвечен
**Ответ:** Да — через `/api/search` + сопоставление `owner/repo`.

## Q8-04: Персистентность результата классификации между запусками
**Приоритет:** Низкое
**Вопрос:** Кэшировать ли вердикт «official vs custom git» по репозиторию, чтобы не запрашивать skills.sh на каждом старте?
**Допущение:** На первой версии повторная классификация опирается на инвариант «наличие git-источника ⟹ custom git; иначе github-lock ⟹ official» (Часть 8, раздел классификации) — отдельный кэш не вводится. Явный кэш — future.
**Статус:** Отвечен
**Ответ:** Да — без отдельного кэша, опираемся на инвариант.

## Заметки по реализации

**Новые/изменённые модули:**
- `src/main/registry/lockAttribution.ts` (новый) — `buildLockAttribution(lock, classifier)`: группирует lock-записи по `sourceUrl`, классифицирует каждый репозиторий один раз (`RepoClassifier`), возвращает `sourcesToEnsure` (git) + `attribution` (slug → git/official). Классификация: SSH/не-github/`sourceType:git` → git; github https → `classifier` (true → official, false → git, null → official).
- `src/main/sources/officialCatalog.ts` — метод `repoPublished(ownerRepo, skillName): Promise<boolean | null>` (true/false/`null` при сетевой/HTTP-ошибке) через `/api/search` + сопоставление `source`.
- `src/main/registry/SkillRegistry.ts` — `setLockAttribution()`; `recomputeOrphans()` заменён на `attributeInstalled()` + `buildInstalledEntry()`/`installedEntry()`: установленный skill без покрытия индексом атрибутируется к official (по карте) или git (по совпадению `config.url`), иначе — локальный сирота. Поля версий переносятся из предыдущей записи. `demoteToLocal(skillId)` — свап official→local (Q8-02). `rescanInstalled()` теперь делегирует в `rebuild()`.
- `src/main/registry/store.ts` — персист множества `demoted` (свапнутые slug'и) рядом с `entries`; `loadDemoted()`.
- `src/main/update/UpdateEngine.ts` — новый dep `officialCatalog`; в `checkEntry`, если official-запись резолвится `unknown`, вызывается `maybeDemoteOfficial` → `repoPublished` == false → `skillRegistry.demoteToLocal`. Свап только при **определённом** отсутствии (не при сетевой ошибке/`null`), чтобы не понижать реальные skills.sh-скилы из-за флуктуаций сети.
- `src/main/index.ts` — `seedSourcesFromLock()`: после `ensureDefaultOfficial()` читает lock, строит атрибуцию (classifier = `officialCatalog.repoPublished`), добавляет недостающие git-источники (`assertNotDuplicate` дедуплицирует), затем `setLockAttribution` + `refreshIndex`. Запускается асинхронно после `init()` — быстрый старт из индекса, затем переклассификация; сетевые ошибки не прерывают старт.

**Ключевые решения:**
- **Инвариант вместо кэша (Q8-04):** отдельная классификация не персистится; git-запись ⟹ custom git, github-lock без git-источника ⟹ official. Персистится только «негативный» оверрайд `demoted` (свап при ошибке).
- **Свап official→local (Q8-02):** оптимистично official при недоступности skills.sh; понижение только на определённом «не найдено» при последующей проверке версий. `demoted` переживает пересборку, поэтому переклассификация не возвращает запись в official.
- **Без сети в горячем пути старта:** `SourceManager.add` только валидирует URL (клонирование — при индексации), поэтому добавление ~N источников дёшево; классификация github-записей — единственные сетевые вызовы, сгруппированы по репозиторию и не блокируют первый рендер каталога.

**Верификация:** typecheck (node+web) 0 ошибок; тесты **154/154** (новые: `lock-attribution.test.ts` — 7 кейсов классификации; `skill-registry.test.ts` — 4 кейса атрибуции/свапа; `official-catalog.test.ts` — 4 кейса `repoPublished`; правка мока `update-engine.test.ts`); build ок; prettier чисто по затронутым файлам.

**Fix (git-источники не появлялись в разделе «Источники»):**
- Сидинг добавляет git-источники в main асинхронно, но renderer грузил список только на старте и по `onSourceIndexed` (сидинг намеренно не индексирует). Добавлена подписка стора `sources` на `onCatalogUpdated` — после сидинга `refreshIndex()` шлёт `catalog:updated`, и список источников перечитывается без перезапуска.
- `buildLockAttribution`: классификация github-репозиториев в skills.sh теперь параллельна, а однозначные git (SSH/не-github/`sourceType:git`) определяются синхронно — медленный/недоступный skills.sh больше не задерживает добавление git-источников. Проверка skills.sh запрашивается по **имени skill** (а не `owner/repo`).
- `officialCatalog.repoPublished`: добавлен таймаут (`AbortSignal.timeout`, 4с), т.к. у `fetch` (undici) нет дефолтного таймаута — иначе зависшее соединение подвешивало сидинг.

**Follow-up:** переклассификация оптимистично-official записей при последующем онлайн-старте (сейчас — только реактивный свап при ошибке); при недостаточной точности `/api/search` — точечный endpoint skills.sh (Q8-03).
