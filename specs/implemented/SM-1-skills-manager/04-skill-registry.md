## Обзор

Skill Registry ведёт локальный индекс skills — единый каталог, собранный из всех подключённых источников после индексации. Индекс используется для быстрого поиска, отображения каталога и проверки обновлений. Registry объединяет данные адаптеров источников (Часть 2), информацию об установленных skills и версии от Version Resolver (Часть 3), и предоставляет каталог с поиском/фильтрацией в UI (Часть 7).

## Пользовательская история

Как пользователь, я хочу видеть единый каталог всех доступных skills из всех источников с их статусом (установлен / доступно обновление) и быстро искать/фильтровать по нему, чтобы не переключаться между репозиториями.

## Критерии приемки

- [x] КП1: Registry строит единый каталог из результатов индексации всех включённых источников.
- [x] КП2: Для каждого skill в каталоге доступны: название, описание, источник, установлен ли, установки по агентам (`installations[]` с версией и путём), последняя версия, признак обновления, дата проверки, статус обновления.
- [x] КП3: Локальный индекс персистится (`registry.json`) и переживает перезапуск (быстрый старт из индекса без переиндексации источников).
- [x] КП4: Поиск по названию и описанию (регистронезависимая подстрока). *(fuzzy/minisearch, Q4-02 — отложено; см. заметки)*
- [x] КП5: Фильтрация по источнику (`sourceIds`).
- [x] КП6: Фильтрация по статусу: установлен / не установлен / доступно обновление.
- [x] КП7: Обнаружение установленных skills на диске по всем агентам (`installedScanner`) + чтение `.skill-lock.json`.
- [x] КП8: Каталог отдаётся из in-memory индекса без сетевых вызовов; пагинация. *(бенчмарк на 500+ — при наличии UI, Часть 7)*
- [x] КП9: Registry обновляет запись после проверки версий (`applyVersion`) и после пересканирования установленных (`rescanInstalled`, вызовет Часть 5). Инкрементальное обновление по источнику через `onIndexed`.

## Поведение

### Затрагиваемые компоненты / модули

- **`src/main/registry/SkillRegistry`** — построение/обновление индекса, слияние данных источников + установленных + версий, запросы поиска/фильтрации.
- **`src/main/registry/store`** — персист индекса (файл в userData); инкрементальное обновление на источник.
- **`src/main/registry/installedScanner`** — обнаружение установленных skills на диске **по всем целевым агентам** (пути каждого агента; для Official — чтение `.skill-lock.json`). Установленность ключуется парой (skill, agent). *(эпик Q-01 — мультиагентность)*
- **`src/main/registry/query`** — поиск/фильтрация/сортировка над индексом.
- **IPC:** `catalog:query` (поиск/фильтры → страница результатов), `catalog:get` (детали skill), `catalog:refreshIndex`; событие `catalog:updated`.

### Логика построения индекса

- Ключ skill: стабильный `id` = `sourceId` + нормализованное имя skill.
- Для каждого включённого источника берётся `RawSkill[]` (Часть 2), сопоставляется с установленными (installedScanner) и версиями (Version Resolver, Часть 3).
- Индексация инкрементальна: обновление одного источника не требует пересбора всего индекса.
- `catalog:query` работает над персистентным индексом (не инициирует сетевые вызовы), обеспечивая мгновенный отклик UI. Версии освежаются отдельно Update Engine (Часть 6) и пишутся в индекс.

### Поиск и фильтрация

- Поиск: по `name` + `description` (регистронезависимо, подстрока/фаззи).
- Фильтр по источнику: множественный выбор `sourceId`.
- Фильтр по статусу: `installed` | `not_installed` | `update_available`.
- Сортировка: по имени, по статусу обновления (опционально по популярности для Official — `installs`).
- Пагинация/виртуализация для сотен записей.

## Пограничные случаи и обработка ошибок

- **Skill в нескольких источниках** (одинаковое имя): показываются как отдельные записи (ключ включает `sourceId`); в UI — группировка/пометка источника.
- **Источник отключён:** его записи скрыты из каталога, но сохранены в индексе (не требуют переиндексации при повторном включении).
- **Установленный skill без источника** (источник удалён): помечается `unknown`/«osиротевший», остаётся видимым как установленный.
- **Пустой каталог:** empty-state в UI.
- **Устаревшие данные версий:** `lastCheckedAt` показывает свежесть; статус `unknown` до первой проверки.
- **Большой каталог:** запрос выполняется над персистентным индексом; виртуализация списка в UI.

## Заметки по реализации

**Расположение:** `src/main/registry/`. IPC: `catalog:query`/`catalog:get`/`catalog:refreshIndex` + событие `catalog:updated`.

**Структура:** `installedScanner.ts` (обход каталогов агентов + `.skill-lock.json`), `store.ts` (`RegistryStore` — персист `registry.json`, атомарно), `query.ts` (`queryCatalog` — фильтр/сорт/пагинация), `SkillRegistry.ts` (оркестратор), `index.ts` (фабрика).

**Ключевые решения:**
- **Разделение построения и версий:** при построении индекса версии НЕ резолвятся (нет сети) — поля `latestVersion/hasUpdate` берутся из предыдущей записи или `null`; `catalog:query` работает мгновенно из in-memory индекса. Версии пишет Update Engine (Часть 6) через `applyVersion(skillId, VersionInfo, checkedAt)`.
- **Инкрементальность:** подписка на `SourceManager.onIndexed` → замена записей только этого источника; version-поля сохраняются по id между пересборками.
- **Установленные:** `installedScanner` сканирует каталоги всех `KNOWN_AGENTS` (включая симлинки — модель CLI), ключ — slug (`normalizeSkillKey`); служебные dot-каталоги пропускаются. `installations[]` per-agent (эпик Q-01).
- **Сироты:** установленные skills без источника → синтетические записи `sourceId='installed'`, всегда видимы; пересобираются при каждом изменении.
- **Отключённые источники:** записи остаются в индексе, но скрыты в `query` (фильтр по enabled-источникам) — переключение не требует переиндексации.
- **Поиск:** подстрока (регистронезависимо) по name+description. Fuzzy (minisearch, Q4-02) отложен — текущий вариант достаточен для MVP; абстракция `queryCatalog` позволяет заменить без изменения потребителей.

**Верификация:** typecheck (node+web) 0 ошибок; тесты — 4 новых (queryCatalog фильтры/сорт/пагинация; SkillRegistry: слияние+сироты, applyVersion→статус, скрытие отключённых), суммарно **29/29**; build ок; **smoke-boot реально построил индекс из установленных skills пользователя** (`~/.claude/skills` → 18 записей + служебные пропущены) и записал `registry.json`; prettier чисто.
