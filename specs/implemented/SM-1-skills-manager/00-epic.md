## Обзор

Разрабатываем кроссплатформенное desktop-приложение (Electron) — единый менеджер установки, обновления и управления skills для LLM-агентов. Приложение использует `skills.sh` / `npx skills` как один из механизмов установки, но не ограничивается им: добавляет собственный трекинг версий, уведомления, автообновление, поддержку нескольких источников (официальный каталог, корпоративные Git-репозитории, локальные каталоги).

Сегодня единого механизма нет — разработчики используют `npx skills`, ручное копирование и т.п. Это лишает прозрачного обновления, уведомлений о новых версиях, централизованного управления источниками и поддержки корпоративных Git-репозиториев.

**Почему декомпозиция.** Это принципиально новое приложение с нуля: 6 самостоятельных архитектурных компонентов, 5+ экранов, расширяемые подсистемы (провайдеры, стратегии версий), несколько независимых режимов проверки обновлений. Части имеют разные жизненные циклы (MVP-провайдеры против future — HTTP/OCI/S3) и могут реализовываться поэтапно и параллельно.

## Пользовательская история верхнего уровня

Как разработчик, использующий LLM-агентов, я хочу единое приложение для подключения нескольких источников skills, просмотра единого каталога, установки независимо от источника и автоматического отслеживания и применения обновлений, чтобы не собирать skills вручную из разрозненных репозиториев и не пропускать новые версии.

## Архитектурный контекст

### Стек

- **Electron** (main / preload / renderer), процессная модель с `contextIsolation: true`, `nodeIntegration: false`, типизированный IPC-мост через `contextBridge`.
- **Сборка:** electron-vite (три пайплайна — main / preload / renderer), electron-builder для дистрибутивов Windows / macOS / Linux.
- **Renderer:** Svelte 5 (runes) + Skeleton UI v4 + Tailwind CSS 4.
- **Тесты:** Vitest (node-окружение для main, jsdom для renderer).
- **Хранилище конфигурации:** персистентный JSON-store (тип `ConfigStore`) в пользовательском каталоге приложения.

### Основные компоненты (модули)

- **Foundation & IPC** — процессный каркас, типизированный IPC-контракт, доменные модели, инфраструктура асинхронных задач и прогресса, конфиг-store. (Часть 1)
- **Source Manager** — управление источниками (skills.sh / Git / локальный каталог): CRUD, вкл/выкл, обновление содержимого, листинг доступных skills, watch локальных каталогов. (Часть 2)
- **Version Resolver** — определение текущей и последней доступной версии через набор расширяемых стратегий. (Часть 3)
- **Skill Registry** — локальный индекс skills, индексация источников, единый каталог, поиск/фильтрация. (Часть 4)
- **Installer Providers** — система провайдеров установки: Official (`npx skills`), Git, Local Folder; авто-выбор по типу источника; исполнение с прогрессом. (Часть 5)
- **Update Engine** — режимы проверки обновлений (вручную / при запуске / по расписанию / по изменению), обновление одного/всех, оркестрация Resolver + Registry + Providers, генерация уведомлений. (Часть 6)
- **UI** — App Shell + навигация, экран Источников, единый Каталог, детали skill, Настройки, центр Уведомлений. (Часть 7)

### Расширяемость (сквозной архитектурный принцип)

- **Installer Providers** и **Version Resolver Strategies** реализуются через реестр с общими интерфейсами. Добавление нового провайдера (HTTP Registry, OCI, S3) или новой стратегии версий не должно требовать изменений в UI и остальной логике — только регистрация новой реализации.
- Все длительные операции (клонирование, индексация, установка, обновление) выполняются асинхронно в main-процессе и стримят прогресс в renderer через событийные IPC-каналы.

## Зависимости между частями

- **Все части** зависят от **Части 1** (Foundation) — IPC-контракт, доменные модели, конфиг-store, инфраструктура прогресса.

- **Часть 4 (Skill Registry)** зависит от **Части 2** (перечень источников и их skills) и **Части 3** (данные о версиях).
- **Часть 5 (Installer Providers)** зависит от **Части 2** (тип источника → выбор провайдера); по завершении установки обновляет индекс Части 4.
- **Часть 6 (Update Engine)** зависит от **Частей 3, 4, 5**.
- **Часть 7 (UI)** зависит от **Части 1** (shell) и потребляет IPC-контракты Частей 2–6; экраны можно разрабатывать против контрактов по мере их готовности.

Рекомендуемый порядок реализации: **1 → (2 ∥ 3) → 4 → 5 → 6 → 7** (UI-заготовки экранов можно вести параллельно, начиная с Части 1).

## Q-01: Целевые агенты установки
**Приоритет:** Блокирующее
**Вопрос:** Для каких LLM-агентов приложение устанавливает skills — только Claude Code, или весь набор, поддерживаемый CLI (Cursor, Codex, OpenCode, Windsurf, Gemini, Copilot, Continue и др.)? Это определяет ключи индекса Registry, пути установки и выбор целевого агента в UI.
**Варианты:**
1. Только Claude Code на MVP; мультиагентность — future.
2. Мультиагентный выбор с первого релиза (как в reference-installer: `-a <agent>`).
**Допущение:** Вариант 1 (Claude Code по умолчанию), с заложенной в модель данных возможностью мультиагентности.
**Статус:** Отвечен
**Ответ:** Вариант 2 — **мультиагентность с первого релиза**. Приложение устанавливает skills для набора агентов, поддерживаемых CLI (`-a <agent>`); пользователь выбирает целевые агенты. Registry ключует установленность по паре (skill, agent); UI даёт выбор агента(ов) при установке и в настройках.
**Производное требование:** при изменении набора выбранных агентов в Настройках выполняется **реконсиляция симлинков** для уже установленных skills — для добавленных агентов симлинки создаются, для снятых удаляются (канонический каталог `.agents/skills` сохраняется). Логика — Часть 5 (`agentReconciler`, `install:reconcileAgents`), триггер и UX — Часть 7 (КП12), обновление индекса — Часть 4.

## Q-02: Модель каталога установки и связь с CLI
**Приоритет:** Блокирующее
**Вопрос:** Использовать каноническую модель CLI (единый каталог `.agents/skills` + симлинки в каталоги агентов, глобальный `~/.agents/.skill-lock.json`), или приложение управляет собственной «директорией установки skills» из настроек (FR9)? От этого зависит, читаем ли мы `.skill-lock.json` CLI как источник истины о версиях или ведём собственный индекс.
**Варианты:**
1. Полностью полагаться на модель CLI: устанавливать через `npx skills`, читать `~/.agents/.skill-lock.json` (`skillFolderHash`) как источник истины, поверх строить свой индекс.
2. Своя директория установки + собственный lock/индекс; CLI используется только как один из провайдеров исполнения.
3. Гибрид: для Official-провайдера — модель CLI и его lock; для Git/Local — собственный индекс и вычисляемые хэши.
**Допущение:** Вариант 3 (гибрид) — соответствует ТЗ («использует skills.sh как один из механизмов, но не ограничивается»).
**Статус:** Отвечен
**Ответ:** **Полагаться на модель CLI как на основную.** Установка через `npx skills` формирует `skillFolderHash` в `.skill-lock.json` — используем его как источник истины о версиях. Там, где модели CLI не хватает (например, кастомные Git-репо, git tags, `CHANGELOG.md`, локальные каталоги) — реализуем **собственный индекс и хэши**, опираясь на доступные исходники CLI (`input/skills-main/`) как референс формата (`skillFolderHash` = GitHub tree SHA; локально — `computedHash` содержимого). То есть: сначала `.skill-lock.json` CLI, поверх — свой слой для источников/механизмов, которые CLI не покрывает (git tags / commit SHA / CHANGELOG). Это ровно распределение обязанностей между Частями 3 (Version Resolver) и 5 (Providers).

## Q-03: Способ запуска CLI (`npx skills`) и офлайн/корпоративная среда
**Приоритет:** Высокое
**Вопрос:** Как приложение исполняет Official-провайдер: всегда `npx skills` (тянет пакет из сети при первом запуске), или бандлит фиксированную версию CLI внутрь приложения? В корпоративной/офлайн-среде `npx` из публичного npm может быть недоступен.
**Варианты:** 1. Всегда `npx skills` (как reference). 2. Бандлить CLI. 3. Настройка: путь к CLI / registry npm в Settings.
**Допущение:** Вариант 1 на MVP, с настройкой пути/registry (Часть 7 Settings) как расширение.
**Статус:** Отвечен
**Ответ:** Проверено в reference-installer: он **ничего не бандлит** (единственные зависимости — React) и всегда запускает `npx -y skills add …`, подтягивая CLI из публичного npm при установке, без фиксации версии. Это простейший путь, но жёстко зависит от доступности публичного npm (проблема в корпоративной/офлайн-среде).
**Как лучше (принятое решение):** гибрид варианта 1 и 3 — по умолчанию `npx skills` из сети, но с двумя улучшениями против reference:
1. **Фиксировать версию CLI** (`skills@<pinned>`), чтобы избежать неожиданных поломок от обновления CLI; версия обновляется осознанно вместе с релизами приложения.
2. В **Настройках (Часть 7)** дать необязательные поля: npm-registry (корпоративное зеркало) и явный путь к бинарю `skills` (для офлайн/предустановленного CLI). Если путь задан — использовать его вместо `npx`.
Бандлинг CLI в приложение не делаем (сложность поддержки, дублирование логики CLI), но настройка registry/пути закрывает корпоративный кейс.

## Q-04: Аутентификация к корпоративным Git-репозиториям и прокси
**Приоритет:** Высокое
**Вопрос:** Как хранить и применять учётные данные для приватных Git-репозиториев (SSH-ключи / HTTPS-токены / `gh`), GitHub API-токен для `skillFolderHash` и настройки прокси (FR9)? Секреты нельзя хранить в открытом виде в конфиг-store.
**Варианты:** 1. Опираться на системный git/ssh-agent + переменные окружения (`GITHUB_TOKEN`, `HTTPS_PROXY`). 2. Хранить токены в OS keychain (safeStorage / keytar). 3. Комбинация: ssh — системный, токены — safeStorage.
**Допущение:** Вариант 3; в конфиг-store хранится только выбор режима (SSH/HTTPS) и адрес прокси, секреты — в OS keychain через `safeStorage`.
**Статус:** Отвечен
**Ответ:** Допущение принято. SSH — через системный git/ssh-agent; токены (HTTPS / GitHub API) — в OS keychain через Electron `safeStorage`. В конфиг-store — только режим (SSH/HTTPS) и адрес прокси, без секретов в открытом виде.

**Пост-ревью (Opus) + сравнение с reference-installer + добор блокеров:**

- Код-ревью: без Critical; исправлены Important (отписки IPC в renderer, `AgentInfo.cliFlag`, сброс пагинации при сортировке, `toasts.guard` на всех действиях) + добор тестов.
- Оптимизации из reference: `cleanCliOutput` (ANSI/спиннеры в логах CLI), `mapWithConcurrency` (параллельный `update.runAll`, лимит 4), `Cache` (TTL/LRU для `/api/search`).
- **Блокер: версии для official-источников** — `buildResolveContext` теперь предпочитает `.skill-lock.json` (реальные `sourceUrl`/`skillPath`/`ref` от CLI), а без lock выводит GitHub-репозиторий из `sourceRef` (`owner/repo@slug`). Раньше `repo.url` = `https://skills.sh` → версии official-скилов не резолвились.
- **Блокер: прокси (Q-04/FR9)** — `src/main/net/proxy.ts` `applyProxy`: глобальный undici-dispatcher для `fetch` (GitHub API / skills.sh) + переменные `HTTPS_PROXY/HTTP_PROXY` (наследуются git/npx). Применяется при старте и при изменении настроек (`config:update` с секцией `network`).
- **Блокер: секреты (Q-04)** — `src/main/secrets/SecretStore.ts` на Electron `safeStorage`: GitHub-токен шифруется на диск (`secrets.bin`), значения в renderer не отдаются (IPC `secrets:` — только available/has/set/delete). Токен прокидывается в `GITHUB_TOKEN`/`GH_TOKEN` при старте и при изменении (наследуется fetch-портами и git). UI — поле в Настройках (write-only, статус «задан/не задан»).

**Добор follow-up (без внешних данных):**

- **[6] `repo.localDir`** — `GitCache.existingDir()` (offline, без сети) прокидывает каталог существующего клона git-источника в `ResolveContext`; включает path-scoped `git log` и чтение CHANGELOG из клона вместо деградации к `ls-remote HEAD`. `UpdateEngine` получает `gitCache` в deps.
- **[10] folder-picker** — IPC `dialog:selectDirectory` (нативный выбор каталога) + кнопка «Обзор…» в форме локального источника.
- **[11] дебаунс + стале-гард** — поиск каталога дебаунсится (250 мс), ответы на устаревшие запросы отбрасываются по `queryId`; `SkillDetail` применяет ответ только если выбранный skill не сменился.
- **[14] тема** — тумблер «Как в системе / Светлая / Тёмная» (Настройки); класс `.dark` на `<html>` через `@custom-variant`, Skeleton подхватывает через `light-dark()`/`color-scheme`; предпочтение в `localStorage`, применяется до монтирования.
- **[15] неиспользуемые IPC-события** — `install:result` → тост об успешной установке, `update:checked` → сводный тост при наличии обновлений (ошибки по-прежнему в уведомлениях).
- **[16] типы источников из реестра** — форма добавления строится из `SOURCE_TYPES`; новый `SourceType` требует правки в одном месте.
- **[17] semver pre-release/build** — `parseSemver` выделяет pre-release, `compareSemver` сравнивает по правилам semver §11 (стабильный релиз старше pre-release); build-метаданные игнорируются.
- **[18] параллелизм** — `resolveConcurrency` (env-override `SKILLS_CHECK_CONCURRENCY`/`SKILLS_INSTALL_CONCURRENCY` + CPU-скейлинг в границах) вместо фиксированных 6/4.

**Follow-up по решениям пользователя:**

- **Безопасность skills (skills.sh)** — обнаружен и подключён API `GET /api/audit?source=<owner/repo>&skills=<slug>`: провайдеры `ath`/`socket`/`snyk`/`zeroleaks`, поле `risk` ∈ safe<low<medium<high<critical (+ score/alerts). `AuditService` (main, TTL-кэш 30 мин, негативы тоже кэшируются) → IPC `catalog:audit`.
