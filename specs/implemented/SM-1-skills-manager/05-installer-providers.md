## Обзор

Механизм установки skills реализуется через систему провайдеров с общим интерфейсом. На MVP: Official Provider (`npx skills`), Git Provider, Local Folder Provider. Провайдер выбирается автоматически по типу источника. Архитектура позволяет добавлять новые провайдеры (HTTP Registry, OCI, S3) без изменений UI и остальной логики — только регистрацией в реестре провайдеров. Установка исполняется асинхронно с прогрессом; по завершении обновляется индекс (Часть 4).

## Пользовательская история

Как пользователь, я хочу устанавливать skill из каталога одним действием независимо от источника, чтобы приложение само выбрало корректный механизм установки и показало прогресс.

## Поведение

### Затрагиваемые компоненты / модули

- **`src/main/installer/registry`** — реестр провайдеров, выбор по типу источника (`resolveProvider(source)`).
- **`src/main/installer/providers`** — `OfficialProvider`, `GitProvider`, `LocalFolderProvider`; общий интерфейс `InstallerProvider`.
- **`src/main/installer/exec`** — обёртка над `JobRunner` для установки (spawn CLI / файловые операции), очистка вывода CLI (ANSI/спиннеры), детекция «already installed».
- **`src/main/installer/paths`** — резолв целевого каталога установки по агенту/scope (эпик Q-01/Q-02).
- **`src/main/installer/agentReconciler`** — реконсиляция симлинков установленных skills при изменении набора агентов (диф `previousAgents` → `nextAgents`: добавить симлинки для новых, удалить для снятых). Работает над каноническим каталогом `.agents/skills` как источником истины; fallback на копирование там, где симлинки недоступны.
- **IPC:** `install:run` (skillId → jobId), `install:cancel`, `install:reconcileAgents` (previousAgents, nextAgents → jobId); события через `job:*` (Часть 1) + `install:result`.

### `InstallerProvider` (общий контракт)

- `id`, `displayName`.
- `supports(source)` — применим ли к типу источника.
- `install(request, jobContext)` — выполнить установку, стримя прогресс/лог через `jobContext`; вернуть `InstallResult`.
- `preflight(request)` — проверка «уже установлен» (без спавна тяжёлых команд, по возможности через файловую систему/lock).

### Логика по провайдерам (текстом, без кода)

- **OfficialProvider:** спавн `npx` (`npx.cmd` на Windows) с аргументами `skills@<pinned> add <source>@<skill>` (версия CLI зафиксирована) и неинтерактивными флагами (`-y`, scope, `-a <agent>` для каждого целевого агента); окружение `CI=1, NO_COLOR=1, npm_config_yes=true`; shell-режим на Windows. Детекция «already installed» из вывода → `skipped`. Lock ведётся CLI (`~/.agents/.skill-lock.json`). **Настройки CLI (эпик Q-03):** если в настройках задан явный путь к бинарю `skills` — использовать его вместо `npx`; если задан npm-registry — прокинуть в окружение спавна. *(reference-installer ничего не бандлит и не фиксирует версию — мы фиксируем версию и даём настройку registry/пути для корпоративной/офлайн-среды.)*
- **GitProvider:** использовать закэшированный клон источника (Часть 2) или клонировать; скопировать/симлинковать папку skill (по `sourceRef`/`skillPath`) в целевой каталог; записать lock-запись (`skillFolderHash` — GitHub tree SHA при доступности API, иначе `computedHash` содержимого). Аутентификация — эпик Q-04.
- **LocalFolderProvider:** скопировать/симлинковать папку skill из локального каталога в целевой; lock-запись с `computedHash`.

### Реконсиляция агентов (при изменении выбора в Настройках)

- Вход: `previousAgents`, `nextAgents`. Вычисляется диф: `added = next \ prev`, `removed = prev \ next`.
- Для каждого установленного skill (из Registry, Часть 4):
  - для каждого агента из `added` — создать симлинк из каталога агента (`~/.claude/skills/<skill>` и т.п.) на канонический `.agents/skills/<skill>`; при недоступности симлинков — скопировать.
  - для каждого агента из `removed` — удалить симлинк/копию из каталога агента. Канонический каталог `.agents/skills` и симлинки агентов из пересечения (`prev ∩ next`) не трогаются.
- Идемпотентность: повторный запуск не создаёт дублей и не падает на уже существующих/отсутствующих ссылках.
- По завершении — обновить `installations[]` в Registry (добавить/убрать записи по агентам), эмитить сводку.

### Пакетная установка

- Поддержка установки нескольких skills с ограничением параллелизма (пул воркеров, авто-масштабирование по CPU, потолок; ср. reference — env `SKILLS_INSTALL_CONCURRENCY`). Обновление «всех» использует ту же машину (см. Часть 6).

## Требования к данным и API

### `InstallRequest`

```json
{
  "skillId": "string",
  "sourceId": "string",
  "sourceRef": "string",
  "targetAgents": "string[] (один или несколько агентов; по умолчанию из настроек)",
  "scope": "string (enum: global | project)",
  "force": "boolean"
}
```

> Мультиагентность (эпик Q-01): установка выполняется для набора агентов. Для Official это `-a <agent>` (по агенту на флаг); для Git/Local — копирование/симлинк в каталог каждого целевого агента. `InstallResult` агрегирует исход по агентам (или возвращается per-agent результат).

### `InstallResult`

```json
{
  "skillId": "string",
  "status": "string (enum: ok | skipped | failed)",
  "installedVersion": "string | null",
  "installPath": "string | null",
  "error": "AppError | null"
}
```

### Целевые пути установки

- Модель путей согласуется с эпик Q-01/Q-02. **Основа — модель CLI** (`.agents/skills` как единый каталог + симлинки в каталоги агентов; глобально `~/.agents/skills`, в проекте `.agents/skills`). Для каждого целевого агента — его путь (`~/.claude/skills`, `~/.cursor/skills` и т.д.). Собственная директория установки из настроек (Часть 7) переопределяет базовый каталог, где это применимо.

## Пограничные случаи и обработка ошибок

- **Уже установлен:** `preflight` → `skipped` (если не `force`).
- **Зависший CLI:** таймаут по бездействию (порог из Части 1) → SIGTERM → SIGKILL, статус `failed`.
- **Отказ auth к приватному Git:** `INSTALL_FAILED` с понятным сообщением; предложить проверить настройки Git (Часть 7).
- **Нет прав на целевой каталог / диск заполнен:** `failed` с диагностикой.
- **Симлинки недоступны (Windows без прав):** fallback на копирование.
- **Прерывание пользователем:** корректная отмена задачи и очистка частично скопированных файлов.
- **Инъекция через имя/URL:** валидация токенов аргументов CLI (safe-token regex, как в reference) для защиты от shell-инъекций.
- **Реконсиляция — снятие всех агентов:** если `nextAgents` пуст, симлинки удаляются из всех каталогов агентов, но канонические файлы `.agents/skills` сохраняются (skill остаётся «установленным в каноне», но не подключён ни к одному агенту) — предупредить пользователя.
- **Реконсиляция — целевой агент не устанавливался ранее / нет канонических файлов:** для такого skill симлинк не создаётся (нечего линковать); фиксируется в сводке как пропуск.
- **Реконсиляция — битые/ручные симлинки в каталоге агента:** перезаписать корректной ссылкой (idempotent), не падать.
- **Official-провайдер и реконсиляция:** для skills, установленных через CLI в каноническую модель (`.agents/skills`), реконсиляция симлинков работает так же; при копийной модели CLI — fallback на копирование/удаление.

## Заметки по реализации

**Расположение:** `src/main/installer/`. IPC: `install:run` / `install:reconcileAgents` (invoke) + событие `install:result`; отмена — через существующий `jobs:cancel`.

**Структура:** `types.ts` (`InstallerProvider`, `ResolvedInstall`), `registry.ts` (`InstallerRegistry`), `paths.ts` (канон `.agents/skills` + пути агентов), `fsLink.ts` (symlink с fallback на копию, идемпотентно), `exec.ts` (сборка `npx`-инвокации + pinned-версия + safe-token + детекция already-installed + spawn с kill по сигналу), `fileInstall.ts` (общий copy→канон + link-агенты для Git/Local), `providers/{official,git,local}.ts`, `agentReconciler.ts`, `InstallerService.ts`, `index.ts` (фабрика).

**Ключевые решения:**
- **Канон-модель (Q-02):** Git/Local копируют skill в `.agents/skills/<name>` (источник истины), затем симлинк в каталог каждого агента (fallback — копия при недоступности симлинков, напр. Windows). Official делегирует установку и lock самому CLI.
- **Official (Q-03):** одна `npx`-инвокация на все агенты (`-a` на каждого), pinned `skills@1.5.14`, `cliPath`/`npmRegistry` из настроек; `CI=1/NO_COLOR=1/npm_config_yes`; safe-token валидация `sourceRef` (анти-инъекция); детекция «already installed» → `skipped`.
- **Мультиагент (Q-01):** `InstallResult.outcomes[]` per-agent; агрегатный статус.
- **Реконсиляция (Q-01):** `agentReconciler` — диф added/removed; посев канона из существующей установки при копийной модели; идемпотентность; канон и пересечение не трогаются. После install/reconcile — `SkillRegistry.rescanInstalled()` (обновляет каталог + эмитит `catalog:updated`).
- **Общий `GitCache`** между Source Manager и Git-провайдером (единый кэш клонов) — создаётся в `main/index` и передаётся обоим.
