## Обзор

Зарегистрировать пользовательский протокол `skill://` и обрабатывать переход по ссылке: открыть/сфокусировать приложение и предложить установку с подтверждением. Ссылка указывает на Git-репозиторий со скилами.

Поддерживаемые форматы:

- `skill://github.com/vercel-labs/agent-skills` → HTTPS-репозиторий `https://github.com/vercel-labs/agent-skills`
- `skill://git@github.com:vercel-labs/agent-skills.git` → SSH-репозиторий `git@github.com:vercel-labs/agent-skills.git`

## Пользовательская история

Как пользователь, я хочу кликнуть по ссылке `skill://…` (в браузере/доке/README) и получить в Skill Sync предложение установить скилы из этого репозитория с подтверждением, чтобы устанавливать в один переход.

## Критерии приемки

- [ ] КП1: Приложение регистрируется как обработчик схемы `skill://` (dev и packaged).
- [ ] КП2: Переход по `skill://…` при закрытом окне — запускает/показывает приложение; при уже запущенном — фокусирует существующий инстанс (single-instance), без второго процесса.
- [ ] КП3: URL корректно парсится в спецификацию Git-источника (HTTPS и SSH-форматы); неподдерживаемый/битый URL → понятная ошибка, без установки.
- [ ] КП4: Показывается подтверждение с разобранными параметрами (репозиторий, режим доступа) до любых изменений.
- [ ] КП5: После подтверждения репозиторий добавляется как Git-источник и индексируется (см. Q-01 о поведении «источник vs прямая установка»).
- [ ] КП6: Отмена подтверждения не производит изменений.

## Поведение

### Затрагиваемые компоненты

- **`src/main/deeplink.ts`** (новый) — регистрация протокола (`app.setAsDefaultProtocolClient('skill')`), single-instance lock (`app.requestSingleInstanceLock`), обработка `open-url` (macOS) и `second-instance`/`process.argv` (Windows/Linux), парсинг URL.
- **`src/main/index.ts`** — инициализация DeeplinkHandler до/после `app.whenReady`; маршрутизация распарсенного deeplink в окно (показать + отправить событие) или к сервису установки.
- **IPC/renderer** — событие «получен deeplink» → окно показывает подтверждение (нативный `dialog:confirm` из SM-1 или in-app модалка, см. Q-02); при подтверждении — добавление источника через существующий `source.add` и переход в каталог, отфильтрованный по источнику.

### Контракты

- Парсер deeplink: вход — строка `skill://…`; выход — `{ type: 'git', url, authMode }` либо ошибка. Правила преобразования:
  - `skill://github.com/<owner>/<repo>` → `url = https://github.com/<owner>/<repo>`, `authMode = https`.
  - `skill://git@<host>:<owner>/<repo>.git` → `url = git@<host>:<owner>/<repo>.git`, `authMode = ssh`.
- Безопасность: применять валидацию токенов/URL, согласованную с SM-1 (`assertSafeToken`/`parseGithubRepo`); не выполнять установку без явного подтверждения.

### Навигация

- Нужна поддержка диплинков — это ядро части. Возврат: после подтверждения — экран каталога с фильтром по добавленному источнику.

## Пограничные случаи и обработка ошибок

1. **Битый/неподдерживаемый URL** — тост/диалог с ошибкой, без изменений.
2. **Источник уже добавлен** — не дублировать (SM-1 `assertNotDuplicate`), показать существующий/предложить открыть.
3. **Второй инстанс** — deeplink передаётся в уже запущенный, окно фокусируется.
4. **Приватный репозиторий** — установка/индексация опирается на системный git (SM-1: SSH/системный credential-helper).

## Зависимости и риски

- Требует single-instance lock — согласовать с трей-поведением SM-1 (окно скрыто, не закрыто).
- Packaged-регистрация протокола зависит от electron-builder (`protocols` в конфиге) — учесть при релизе.

## Заметки по тестированию

- Юнит: парсер deeplink (HTTPS/SSH/битые/edge).
- Ручной: переход по ссылке при закрытом и открытом окне (все ОС по возможности), отмена/подтверждение.

## Открытые вопросы

## Q-01: Поведение перехода — добавить источник или сразу установить

**Приоритет:** Блокирующее
**Вопрос:** URL указывает на репозиторий (без конкретного скила). Что делает переход после подтверждения?
**Варианты:**
1. Добавить репозиторий как Git-источник, проиндексировать и открыть каталог для выбора скилов (рекомендуется — соответствует модели SM-1 «источник → каталог → установка»).
2. Добавить источник и сразу установить все скилы репозитория в целевые агенты по умолчанию.
3. Трактовать `owner/repo` как official (skills.sh) и установить.
**Допущение:** Вариант 1.
**Статус:** Отвечен
**Ответ:** Вариант 1 — добавить репозиторий как Git-источник, проиндексировать и открыть каталог для выбора скилов.

## Q-02: UI подтверждения — нативный диалог или in-app модалка

**Приоритет:** Высокое
**Вопрос:** Показывать подтверждение нативным `dialog:confirm` (как в SM-1) или отдельной in-app модалкой с деталями репозитория?
**Допущение:** Нативный `dialog:confirm` в первой версии.
**Статус:** Отвечен
**Ответ:** Нативный `dialog:confirm` (как в SM-1).

## Q-03: Ограничение хостов

**Приоритет:** Среднее
**Вопрос:** Ограничивать ли поддерживаемые хосты (только github.com) или допускать любой git-хост (GitLab и пр.)?
**Допущение:** Разрешать любой валидный git-хост; парсинг HTTPS/SSH общий, не только github.com.
**Статус:** Отвечен (допущение принято)
**Ответ:** Любой валидный git-хост.
