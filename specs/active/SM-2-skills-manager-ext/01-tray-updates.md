## Обзор

Расширить меню трея (SM-1 `src/main/tray.ts`) динамическим подменю «Обновления» со списком скилов, для которых доступно обновление, и пунктом «Обновить все». Позволяет обновлять скилы, не открывая окно.

## Пользовательская история

Как пользователь, я хочу видеть в трее скилы с доступными обновлениями и обновлять их одним кликом (по одному или все сразу), чтобы держать скилы актуальными без открытия приложения.

## Критерии приемки

- [ ] КП1: В меню трея есть подменю «Обновления» со списком скилов, у которых `updateStatus = update_available`.
- [ ] КП2: Клик по элементу подменю запускает обновление этого скила (аналог `update.runOne`).
- [ ] КП3: Пункт «Обновить все» запускает обновление всех обновляемых (аналог `update.runAll`).
- [ ] КП4: При отсутствии обновлений подменю неактивно (disabled) или показывает «Нет обновлений».
- [ ] КП5: Список в подменю обновляется после проверки обновлений и после применения (без перезапуска приложения).

## Поведение

### Затрагиваемые компоненты

- **`src/main/tray.ts`** — меню строится динамически; добавляется подменю «Обновления» (submenu) и пункт «Обновить все». Модуль получает провайдер актуального списка обновляемых скилов и колбэки `updateOne(skillId)` / `updateAll()`.
- **`src/main/index.ts`** — при создании трея передаёт провайдер (запрос к `SkillRegistry` по `updateStatus = update_available`) и колбэки (`UpdateEngine.runOne/runAll`). Подписывается на события завершения проверки/каталога, чтобы пересобирать меню трея.

### Контракты

- Провайдер списка: возвращает элементы `{ skillId, name, installedVersion, latestVersion }` для скилов с доступным обновлением (из текущего состояния `SkillRegistry`, без сетевых вызовов).
- Пересборка меню: инициируется по событиям обновления каталога/результата проверки (переиспользовать существующие сигналы SM-1).

### Навигация

- Диплинки — не требуются (см. Часть 2).

## Пограничные случаи и обработка ошибок

1. **Нет обновлений** — подменю disabled / «Нет обновлений».
2. **Трей недоступен** (SM-1: `createTray` вернул null) — фича неактивна, ошибок нет.
3. **Много обновлений** — список может быть длинным; при необходимости ограничить верхним N с пунктом «Открыть приложение» (см. Q-01).

## Заметки по тестированию

- Юнит: формирование списка обновляемых из состояния `SkillRegistry`.
- Ручной: проверить, что подменю обновляется после `Проверить обновления` и после применения обновления.

## Открытые вопросы

## Q-01: Ограничение длины подменю

**Приоритет:** Низкое
**Вопрос:** Ограничивать ли число скилов в подменю (например топ-N + «Показать все в приложении»)?
**Допущение:** Без ограничения в первой версии; при >N (например 15) добавить пункт «Открыть приложение».
**Статус:** Отвечен (допущение принято)
**Ответ:** Без жёсткого ограничения; при большом числе — пункт «Открыть приложение».

## Q-02: Индикатор количества обновлений на иконке трея

**Приоритет:** Низкое
**Вопрос:** Показывать ли число доступных обновлений в тултипе трея / бейджем на иконке (где ОС поддерживает)?
**Допущение:** Отражать количество в тултипе трея; бейдж — future.
**Статус:** Отвечен (допущение принято)
**Ответ:** Количество — в тултипе трея; бейдж на иконке — future.

## Заметки по реализации

- Реализовано в виде класса `AppTray` (`src/main/tray.ts`), который принимает зависимости `getUpdatableSkills`, `updateOne`, `updateAll`.
- Вызов метода `rebuild()` обновляет содержимое контекстного меню на основе списка скиллов с доступными обновлениями и изменяет tooltip трея, отображая количество обновлений.
- В `src/main/index.ts` добавлено обращение к `tray?.rebuild()` при событиях `catalogUpdated` и `updateChecked`.
- Применено ограничение в 15 элементов для отображения списка обновляемых скиллов в подменю, если элементов больше - добавляется пункт "И ещё N... (открыть приложение)", что соответствует Q-01.
- Tooltip содержит количество доступных обновлений, что соответствует Q-02.
- Ревью: тесты, build, lint успешно пройдены.
