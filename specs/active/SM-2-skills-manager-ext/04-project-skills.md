## Обзор

Крупная новая фича: возможность устанавливать и отслеживать скилы в конкретных **проектах** (проектный scope), а не только глобально (`~`). Пользователь ведёт список отслеживаемых проектов (каталогов), приложение сканирует и показывает установки по каждому проекту — по аналогии с текущим глобальным трекингом (`installedScanner` по `home`).

Основано на SM-1: проектный scope уже частично заложен (`InstallScope = 'global' | 'project'`, `AgentInfo.projectDir`, канонический `.agents/skills`), но `project` использует `process.cwd()` (каталог запуска приложения), что для десктоп-приложения не имеет смысла. Эта часть заменяет «cwd» на явно выбранные проекты и вводит трекинг по проектам.

**Ключевой архитектурный принцип (DRY):** источники (git/local/official) — **общие** для глобального и проектного scope; различаются только *установки* (на диске). Поэтому индекс записей каталога (`SkillRegistry.entries`) строится один раз из источников, а карт установок несколько: глобальная + по одной на каждый отслеживаемый проект. И бэкенд-запрос каталога, и UI-компоненты параметризуются «контекстом scope», а не дублируются.

## Пользовательская история

Как пользователь, я хочу добавить конкретные проекты (папки) в Skill Sync и устанавливать/обновлять/переустанавливать скилы отдельно для каждого проекта, видя, что где установлено, чтобы управлять project-scoped скилами наравне с глобальными.

## Критерии приемки

- [ ] КП1: Пользователь может добавить проект (выбор папки через нативный `dialog:selectDirectory` из SM-1) и удалить его из списка; список сохраняется в конфигурации.
- [ ] КП2: Приложение сканирует установленные скилы в каждом отслеживаемом проекте (каталоги агентов относительно корня проекта + канонический `.agents/skills`) и показывает их отдельно от глобальных.
- [ ] КП3: Установка скила может быть нацелена на конкретный проект (проектный scope с путём проекта) или глобально.
- [ ] КП4: Обновление и переустановка (Часть 3) работают в контексте выбранного проекта.
- [ ] КП5: Реконсиляция набора агентов (SM-1 Q-01) применяется к выбранному проекту, а не к `cwd`.
- [ ] КП6: Удаление проекта из трекинга не трогает файлы на диске (только перестаём отслеживать).
- [ ] КП7: Раздел «Проекты»: каждый проект отображается как **каталог** (иконка папки, имя, путь), но явно читается как отдельный проект; список скилов проекта открывается «внутри» проекта.
- [ ] КП8: Каталог, детали скила, фильтры, поиск и группировка в проектном контексте — те же компоненты, что и в глобальном (единый код, разный контекст); источники — те же самые.
- [ ] КП9: Пользователь может одним переключателем добавить/убрать папки скилов проекта в `.gitignore` этого проекта; операция идемпотентна и не трогает пользовательские строки файла.

## Поведение

### DRY: переиспользование каталога и деталей

Цель — не дублировать экран каталога и деталей для проектов, а параметризовать существующие «контекстом scope».

- **Тип контекста** (`src/shared/domain/scope.ts`, новый): `ScopeContext` — дискриминированное объединение `{ scope: 'global' }` | `{ scope: 'project'; projectId: string; projectPath: string }`. Единый объект, который пробрасывается из UI в IPC и далее в `PathContext`/`CatalogQuery`.
- **`CatalogStore` → инстанс на контекст** (`src/renderer/src/lib/stores/catalog.svelte.ts`): класс уже инкапсулирует состояние поиска/фильтров/результата. Делаем его конструируемым с `ScopeContext` (по умолчанию — `global`, текущий синглтон `catalog` сохраняется как глобальный экземпляр). Для открытого проекта создаётся project-экземпляр. Запрос `api.catalog.query` получает `context`.
- **Презентационный компонент `SkillCatalog.svelte`** (новый) — извлекается из текущего `CatalogPage.svelte`: панель фильтров/поиска/группировки + список (виртуализация) + карточка скила. Принимает пропсы `store: CatalogStore`, `context: ScopeContext`, коллбэк открытия деталей. Текущий `CatalogPage.svelte` становится тонкой обёрткой над глобальным `store`; проектный экран рендерит тот же `SkillCatalog.svelte` с project-`store`.
- **`SkillCard.svelte`** (новый) — выносим сниппет `skillCard` из `CatalogPage.svelte` в отдельный компонент, чтобы им пользовались оба контекста (и потенциально страница проекта).
- **`SkillDetail.svelte`** — параметризуется контекстом: отображает установки/пути и целевых агентов относительно scope (для проекта — относительно корня проекта). В проектном контексте действия установки/обновления/переустановки/удаления идут в этот проект.
- **`installWithAuditGuard` / `uninstallWithConfirm`** (`src/renderer/src/lib/install.ts`) — добавляется параметр цели (`ScopeContext`), по умолчанию — глобальный (текущее поведение не меняется). Проектные экраны передают project-контекст; `InstallRequest` заполняется из него.

### Раздел «Проекты» (проект как каталог)

- **Навигация**: новый пункт «Проекты» в левом меню (`nav` в `App.svelte`), иконка папки. `View` (`ui.svelte.ts`) расширяется значением `projects`; в `UiStore` добавляется `projectId: string | null` (какой проект открыт). Модель навигации: список проектов ↔ «вход» в проект (каталог скилов проекта). Хлебные крошки «Проекты / <имя проекта>» для выхода назад.
- **`ProjectsPage.svelte`** (новый) — список отслеживаемых проектов. Каждый проект — карточка в виде **каталога**: иконка папки, имя проекта (по умолчанию — имя папки) как «название каталога», полный путь моноширинным приглушённым текстом под именем, бейдж с числом установленных в проекте скилов. Действия: открыть проект (переход внутрь), открыть в файловом менеджере (`shell.openPath`), удалить из трекинга (нативное подтверждение; файлы не трогаем). Пустое состояние с кнопкой «Добавить проект» (нативный выбор папки).
- **`ProjectView.svelte`** (новый) — экран открытого проекта. Шапка: хлебные крошки + иконка папки + имя + путь + действия (открыть в ФС, переключатель `.gitignore`, удалить). Тело: `SkillCatalog.svelte` с project-контекстом. Визуально проект «выглядит как каталог», но шапка/крошки явно сообщают, что это конкретный проект.

### Стратегия сканирования проектов (см. Q-07)

- Установки проектов сканируются **при запуске** приложения и **по кнопке «Обновить»** — карточки списка сразу показывают число установленных скилов.
- При большом числе проектов eager-скан ограничивается **N самыми свежими** (порог-константа); «старые» проекты сканируются **лениво** при открытии.
- «Свежесть» проекта — время последнего git-коммита в корне (фолбэк на `mtime`, если не git-репозиторий); helper на стороне main. Результаты кэшируются в `SkillRegistry`.

### Общие источники (то же, что и глобально)

- Индекс `SkillRegistry.entries` строится из тех же источников (`SourceManager`), что и для глобального каталога — отдельного набора источников для проектов нет. Проектный каталог показывает **те же скилы**; отличается только состояние установки и цель установки.
- Поиск, фильтры (статус/источник), сортировка, группировка по источнику — тот же компонент и та же логика `queryCatalog`; статус-фильтры (`installed`/`not_installed`/`update_available`) вычисляются относительно контекста запроса.

### Управление `.gitignore` проекта

- **`GitignoreService`** (main, новый) — читает/пишет `<project>/.gitignore`, управляя выделенным маркированным блоком:

  ```
  # >>> Skill Sync (managed) >>>
  .agents/skills/
  .claude/skills/
  # <<< Skill Sync (managed) <<<
  ```

  - Содержимое блока — канонический `.agents/skills/` плюс проектные каталоги (`agentDir(agent, 'project')`) агентов, для которых в проекте есть установки/цель. Дедуп путей (универсальные агенты уже покрыты `.agents/skills/`).
  - Идемпотентно: при повторном применении переписывается только управляемый блок между маркерами; пользовательские строки не изменяются. Отключение — удаление блока целиком.
  - Файла нет → создаётся. Проект не под git → операция всё равно допустима (просто пишем файл), но UI подсказывает, что репозиторий не обнаружен.
- **Домен `TrackedProject`** получает поле `gitignoreManaged: boolean` (сохраняется в конфиге). При включённом флаге установка нового скила в проект также освежает управляемый блок (учитывает новые затронутые каталоги агентов).
- **UI**: переключатель «Добавить папки скилов в `.gitignore`» в шапке `ProjectView` (и предлагается после первой установки в проект). Состояние читается через `project:gitignore` status.

### Затрагиваемые компоненты / экраны

- **Домен `TrackedProject`** (`src/shared/domain/project.ts`, новый) — модель отслеживаемого проекта (+ `gitignoreManaged`).
- **Домен `ScopeContext`** (`src/shared/domain/scope.ts`, новый) — единый контекст scope для UI/IPC/путей.
- **`AppConfig`** — новое поле `projects: TrackedProject[]`; дефолт — пустой список (миграция схемы не требуется: `ConfigStore.load` доливает отсутствующие секции из `defaultConfig()`).
- **`ProjectStore`** (main, новый) — CRUD отслеживаемых проектов поверх `ConfigStore`, IPC-обёртки.
- **`GitignoreService`** (main, новый) — управляемый блок в `.gitignore` проекта.
- **`AgentInstallation`** (`src/shared/domain/skill.ts`) — расширяется контекстом расположения: `scope: 'global' | 'project'` и `projectId: string | null`.
- **`installedScanner`** — обобщается: общий обход каталогов агентов извлекается в helper, вызываемый и для `home` (global), и для корня каждого проекта; результат помечается scope/projectId.
- **`SkillRegistry`** — держит один `entries` (источники) + карту установок по контексту (`Map<ctxKey, InstalledMap>`, `ctxKey` = `global` | `project:<id>`); запрос проецирует установки на запрошенный контекст (см. Q-02, Q-10).
- **`paths.ts` / `InstallerService`** — `PathContext` для project scope использует **корень выбранного проекта** (`ctx.cwd = project.path`) вместо `process.cwd()`; `InstallRequest` получает целевой проект.
- **UI** — новый раздел «Проекты» (`ProjectsPage`, `ProjectView`), извлечённые `SkillCatalog.svelte`/`SkillCard.svelte`, параметризованные `CatalogStore`/`SkillDetail`.

### Контракты

- **`TrackedProject`** (JSON Schema):

```json
{
  "id": "string (UUID)",
  "path": "string (абсолютный путь к корню проекта)",
  "name": "string (отображаемое имя; по умолчанию — имя папки)",
  "gitignoreManaged": "boolean (управляем ли блок .gitignore; по умолчанию false)"
}
```

- **`ScopeContext`**: `{ scope: 'global' }` | `{ scope: 'project'; projectId: string; projectPath: string }`.
- **`AgentInstallation`** (расширение): добавляются поля `scope` (`global` | `project`) и `projectId` (`string | null`).
- **`CatalogQuery`** (расширение): необязательное поле `context?: ScopeContext` (по умолчанию — global). Влияет на вычисление `installed`/`installations`/`updateStatus` записи.
- **Установка в проект**: `InstallRequest` с `scope = 'project'` и идентификатором/путём целевого проекта; `PathContext.cwd = project.path`.
- **Сканирование проекта**: для каждого `TrackedProject` — обход каталогов агентов (`agentDir(agent, 'project')` относительно `project.path`) и канонического `.agents/skills`; результат помечается `scope='project'`, `projectId`.
- **IPC namespace `project:*`**: `list()`, `add(path)`, `remove(id)`, `rename(id, name)`; `gitignore.status(id)`, `gitignore.apply(id)`, `gitignore.remove(id)`.

### Навигация

- Новый раздел «Проекты» в навигации приложения (список → вход в проект → каталог скилов проекта; хлебные крошки для возврата).
- Добавление проекта — через нативный выбор папки.
- Диплинки — не требуются здесь (Часть 2 может позже нацеливаться на проект — follow-up).

## Требования к данным и API

### Доменная логика

- Новый `ProjectStore` (CRUD проектов) + IPC namespace `project:*` (list/add/remove/rename + gitignore).
- Расширение сканера и реестра установок контекстом scope/project (единый индекс записей, карты установок по контексту).
- Версионные проверки: применяются к скилу (по источнику) и отражаются на всех его установках; обновление/переустановка выполняются пер-установка (глобально и/или в конкретных проектах) — см. Q-03.
- `GitignoreService`: генерация управляемого блока из набора затронутых каталогов агентов + канона.

## Пограничные случаи и обработка ошибок

1. **Проект удалён с диска** — помечать недоступным, не падать при сканировании; предложить убрать из списка.
2. **Дубликат проекта** — не добавлять один и тот же путь дважды (нормализация абсолютного пути).
3. **Один скил в нескольких местах** — корректно показывать множественные установки (global + N проектов) без «схлопывания».
4. **Пустой список проектов** — раздел «Проекты» с пустым состоянием и кнопкой «Добавить проект».
5. **Права доступа к каталогу проекта** — обработать ошибку чтения/записи, понятное сообщение.
6. **`.gitignore` отсутствует / нет прав на запись** — создать файл; при ошибке записи — понятный тост, флаг `gitignoreManaged` не выставляется.
7. **Проект не является git-репозиторием** — управление `.gitignore` разрешено, но UI показывает подсказку, что `.git` не найден.
8. **Ручные правки внутри управляемого блока** — при повторном `apply` блок перезаписывается (пользовательские правки внутри маркеров не сохраняются); строки вне маркеров не трогаются.

## Q-02: Представление установок global vs project в каталоге

**Приоритет:** Блокирующее
**Вопрос:** Как показывать установки разного scope? Один каталог с бейджами расположения у скила, или переключатель контекста «Глобально / Проект X», или отдельная страница проекта со своим списком?
**Варианты:**
1. Отдельная страница «Проекты» → выбор проекта → список его скилов (в дополнение к глобальному каталогу). Установки скила отражаются с указанием мест (global + проекты).
2. Переключатель контекста в каталоге (Глобально / конкретный проект), меняющий scope всех операций.
**Допущение:** Вариант 1 (страница проектов + пометки мест установки в деталях скила).
**Статус:** Отвечен
**Ответ:** Вариант 1 — отдельная страница «Проекты» → выбор проекта → список его скилов; в деталях скила отражаются места установки (global + проекты).

## Q-03: Гранулярность обновления/переустановки по местам установки

**Приоритет:** Высокое
**Вопрос:** Обновление/переустановка применяются ко всем местам установки скила сразу или выбранно (только global / только конкретный проект)?
**Допущение:** Пер-место: действие выполняется в том контексте, откуда запущено (страница проекта → в этот проект; глобальный каталог → глобально). «Обновить все» в трее — по глобальным (проектные — из страницы проекта).
**Статус:** Отвечен
**Ответ:** Пер-место (по контексту запуска); «Обновить все» в трее — по глобальным установкам.

## Q-07: Когда сканировать проекты

**Приоритет:** Среднее
**Вопрос:** Сканировать установленные скилы проектов лениво (при открытии проекта) или сразу при запуске (чтобы показывать число скилов в карточках списка)?
**Статус:** Отвечен
**Ответ:** Гибрид с приоритизацией по давности:
- По умолчанию проекты сканируются **при запуске приложения и по кнопке «Обновить»** (eager), чтобы карточки списка сразу показывали число установленных скилов.
- Если проектов «слишком много» (порог — константа, напр. `PROJECT_EAGER_SCAN_LIMIT`), eager-скан ограничивается **N самыми «свежими» проектами**; остальные («старые») сканируются **лениво** — при открытии проекта.
- «Свежесть» проекта определяется по git-истории: время последнего коммита в корне проекта (напр. `git log -1 --format=%ct`), с фолбэком на `mtime` каталога/`.git`, если это не git-репозиторий. Проекты сортируются по этому времени по убыванию; первые N — eager, остальные — lazy.
- Результаты кэшируются в `SkillRegistry` (карты установок по контексту, см. Q-10); карточки «старых» проектов показывают число фоновой дозагрузкой при первом открытии.
**Детали реализации:** helper определения «свежести» проекта (git last-commit time с фолбэком на mtime) — на стороне main (напр. в `ProjectStore` или отдельном `projectActivity.ts`); порог eager-скана вынести в константу.
